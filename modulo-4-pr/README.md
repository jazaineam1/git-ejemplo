# 🔀 Módulo 4: Pull Requests

## 🎯 Objetivo

Aprender a entregar cambios como se hace en la industria: mediante revisión, contexto y trazabilidad.

## 🌍 Contexto

Hacer `push` no significa que el trabajo ya quedó integrado al proyecto principal. En equipos profesionales, antes de unir cambios a `main`, normalmente se pasa por un proceso de revisión.

Ese proceso ocurre mediante un Pull Request, también conocido como PR.

## 🧠 ¿Qué es un Pull Request?

Es una solicitud formal para integrar una rama dentro de otra, normalmente hacia `main`.

Con un PR estás diciendo:

"Terminé este cambio. Revísenlo y, si está correcto, únanlo al proyecto principal."

## 🔁 Flujo real

1. Crear rama
2. Hacer cambios
3. Hacer commits
4. Hacer push
5. Abrir Pull Request en GitHub
6. Recibir comentarios
7. Ajustar si hace falta
8. Integrar el cambio

## 💬 Qué debe tener un PR

### Título claro

Debe resumir el cambio principal.

Ejemplos:

- `Agrega sección de perfil profesional`
- `Corrige error en README del módulo 2`

### Descripción útil

La descripción debe responder:

- qué hiciste
- por qué lo hiciste
- qué archivos o partes del proyecto cambiaste
- cómo se puede revisar o probar

### Contexto

No asumas que el revisor adivina todo. Si el cambio responde a un ejercicio, error o mejora, dilo.

## 👀 Qué revisa normalmente otra persona en tu PR

- si el cambio resuelve lo que promete
- si el código o contenido es claro
- si el alcance está bien delimitado
- si no se dañó algo existente
- si la documentación acompaña el cambio

## 🧑‍🏫 Idea importante

En la vida real, muchas veces lo que se evalúa no es solo si "funciona", sino si el cambio se entiende, se puede revisar fácil y está bien comunicado. Por eso este módulo es clave.
